# Unbounded 引擎架构宣言

> 版本：v1.0
> 性质：定位文档，不是教程。回答"我们是什么"，不回答"怎么写代码"。
> 修订规则：只有当某条契约被证明"挡路"（真实造成了一次不得不违反它才能推进的情况）时才修改本文件，且必须在下方"修订记录"留痕，不做无理由的措辞打磨。

---

## 0. 名字

**问题驱动分层引擎（Problem-Driven Layered Engine，内部简称 PDLE）**

不是 ECS，不是纯 Command 模式，不是任何一本书里能直接翻到的名字。这是从具体问题里长出来的架构，命名是为了让它有身份，不是为了往它头上贴一个更响亮的标签。

## 1. 这是什么范式，不是什么范式

**是**：一组自下而上、按"改一个系统会不会牵连一大片"这条尺子筛出来的分层契约。每一层解决它自己的问题，不预设未来会长成什么样子。

**不是**：不是先画好整体架构图再填代码的自顶向下设计。也不是照搬某个通用框架（ECS/Command/MVC）来"提前购买"未来可能用不上的扩展性。教科书架构解决的是"未知的、别人的"规模问题；这里解决的是"已知的、自己的"具体问题。

**判断新代码该走哪条路的问题**：这段代码是在解决一个已经真实发生的具体问题，还是在为一个还没发生的假想问题打地基？前者留下，后者先不做。

## 2. 核心契约（六条，违反即视为架构损伤）

1. **Tile / Entity 二分**：Tile 是背景，几乎不变；Entity 是内容（树/怪物/NPC），会变化。任何新功能先问它属于哪一类，不允许模糊处理。
2. **六层单向依赖管线**：Seed → Physics → Geology → Climate → Ecology → Civilization → History。不可跳层，不可反向。上层可以读下层的结果，下层永远不知道上层的存在。
3. **EventBus 只广播，不决策**：谁订阅、订阅后做什么决定，是订阅者的责任；EventBus 本身不允许出现业务判断逻辑。
4. **数据优于代码**：新增游戏内容（物品/怪物/配方/文明）优先通过 JSON 表达；只有规则本身变化才动代码。
5. **UI/输入层与游戏逻辑层严格分离**：该不该发生这个动作，是逻辑层（如 player_action.py）的职责；怎么触发它，是输入层（如 play_state.py）的职责。输入层不持有游戏状态的决策权。
6. **Turn 是唯一的时序权威**：任何 Action、任何 System，都不能自行决定"这一步算不算一个回合"。Action 只声明意图和代价，回合何时推进，只有一个决策点。

## 3. 架构健康度量（这是我们自己发明的，教科书里没有）

**唯一问题**：改动一个系统，需不需要牵动十几个文件？

- 需要 → 说明某处已经变成了隐性枢纽（例如曾经的 legacy_system.py），排优先级去解耦。
- 不需要 → 健康，不要因为"这不是经典模式"而主动重构。

这条度量优先于对"代码美不美"的直觉判断。美观但没有牵连问题的代码不需要动。

## 4. 明确不吸收的东西（以及为什么）

- **不引入通用 Command/Action 总线架构**：曾评估过一版完整的 Action 模式重构方案，问题在于它会让 Action 层悄悄接管 Turn 的推进权（每个 handler 自己调 advance_turn），这违反契约 #6。如果未来真的需要类似机制，只允许以"声明式 turn_cost 字段 + 唯一推进点"的最小形式引入，不允许引入完整的 Command 对象体系。
- **不做预防性的 EventBus 优先级/阶段化改造**：除非出现具体的、可复现的顺序错乱 bug，否则契约 #3 已经够用。
- **不做原型/模板 ID 的热重载体系**：这是数据驱动契约（#4）的自然延伸方向，但只有在真实遇到"改数据必须重启进程"的具体痛点时才启动，不提前建。

## 5. 这份文档解决的问题

不是给代码加新能力，是把六条已经在实际决策中被反复使用的原则，从"STATUS.md 里的口头共识"变成"有名字、可以被指着说'这违反了我们的架构'的正式契约"。下次任何一份外部方案（无论来自哪个 AI 或哪本书）声称要引入某个经典模式，先对照第 4 节和第 2 节，而不是先判断这个模式本身好不好。
